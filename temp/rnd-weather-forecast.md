# Weather Forecast & Alerts System Alignment Review

This document evaluates the alignment between the **Weather Forecast**, **Weather Warning (Raw API)**, and **Weather Alert (Normalized Database)** implementations in Emergency OS.

---

## Executive Summary: Is it aligned?

**No, the current system is not fully aligned.** 

While the provider pattern (`getEmergencyData({ mode })`) represents the correct target architecture for a switchable and unified data model, the weather components currently suffer from **architectural drift**, **double-sourcing**, and **incomplete coverage**.

### ⚠️ Primary Issues Identified:
1. **Weather Forecasts are completely missing from the Switchable Provider Layer.**
   * The `WeatherForecast` structure exists and is displayed in the UI, but it is bypassed around the provider. It is called directly via external APIs on both the production Home page (`app/page.tsx`) and the `test-ui` Hub page (`app/test-ui/page.tsx`).
   * Because of this, **Weather Forecasts cannot be simulated** (e.g. testing the UI during a simulated severe Kelantan flood vs a Penang dry spell).
2. **Double-Sourcing of Weather Alerts/Warnings in Live Mode.**
   * In `live` mode, the provider fetches **both** normalized database-backed `weatherAlerts` AND raw, live API-fetched `weatherWarnings`.
   * The `test-ui` dashboard page further splits this: it ignores the database-driven `data.weatherAlerts` in live mode and instead maps raw live warnings on-the-fly. This negates the benefit of our database-first ingestion pipeline.
3. **No Database Persistence or Ingestion for Weather Forecasts.**
   * While we ingest shelters and weather alerts through the `sync-live-data` Supabase Edge Function, we do not store current weather forecasts. Thus, no historical timelines or fallback caches exist for weather forecasts if the MET Malaysia endpoint degrades.

## Detailed Gap Analysis & Code Walkthrough

### 1. The Weather Forecast "Silo"
In `types/emergency.ts`, `EmergencyDataSnapshot` defines the unified contract:
```typescript
export interface EmergencyDataSnapshot {
  mode: DataMode;
  scenarioName?: string;
  shelters: PPS[];
  weatherWarnings: WeatherWarning[];
  weatherAlerts: WeatherAlert[];
  savedLocations: SavedLocation[];
  // ... but where is weatherForecasts?
}
```
Because `weatherForecasts` is excluded:
* The production page (`app/page.tsx`) must fire off a direct external API fetch during SSR:
  ```typescript
  const [alerts, weatherWarnings, weatherForecasts] = await Promise.all([
    getAlerts(),
    getWeatherWarnings(),
    getWeatherForecasts(), // Raw direct API fetch
  ]);
  ```
* The prototype dashboard (`app/test-ui/page.tsx`) does the same, manually converting `SavedLocation` to `WeatherForecastLocation` to feed the `WeatherForecastWidget`.
* **The Gaps**:
  * We cannot test the `WeatherForecastWidget` under simulation scenarios.
  * We are making redundant, un-cached external fetches on page render.

### 2. Dual Data Tracks for Alerts in `test-ui`
In `app/test-ui/page.tsx`, we see on-the-fly mapping that bypasses the database:
```typescript
  const isLive = mode === "live";
  const displayAlerts = isLive
    ? weatherWarnings.map((w, index) => ({
      id: `live-warning-${index}`,
      title: w.warning_issue?.title_en || w.heading_en || "Weather Alert",
      description: w.text_en || w.text_bm || "",
      severity: "warning" as const,
      affectedArea: w.warning_issue?.title_en || "Malaysia",
    }))
    : data.weatherAlerts;
```
* **The Gaps**:
  * When `mode === "live"`, the UI displays alerts generated by fetching the MET API on-the-fly.
  * In contrast, the personalized notification generator (`checkEmergenciesAndAlertsForLocation` inside `app/profile/actions.ts`) scans the `weather_alerts` **database table** which is synced asynchronously.
  * This creates a split-brain bug: the dashboard's "Weather Alerts" widget may display a live alert, but because it hasn't been ingested into the `weather_alerts` table yet (or ingestion failed), no personal in-app notifications are shown for that alert.

### 3. Raw Warnings vs. Normalized Alerts
The provider returned snapshot contains both:
* `weatherWarnings: WeatherWarning[]` (raw API structure)
* `weatherAlerts: WeatherAlert[]` (normalized database model)

In `data/providers/live-provider.ts`, getting the live snapshot does this:
```typescript
  const [..., weatherAlertResult, weatherWarnings, ...] = await Promise.all([
    supabase.from("weather_alerts").select("*")..., // Normalized from DB
    getWeatherWarnings(),                           // Raw live API fetch
    ...
  ]);
```
* **The Gaps**:
  * Every page render queries the database and *also* makes a live REST call to `api.data.gov.my`.
  * The raw `weatherWarnings` feed should ideally be obsolete for the UI once ingestion is mature. The UI should only consume normalized `weatherAlerts`.

## Realignment & Remediation Plan

To establish alignment with the target architecture defined in the system documentation, we propose the following systematic refactoring plan.

### Phase 1: Bring Weather Forecasts into the Provider Model
1. **Extend Contract (`types/emergency.ts`)**:
   Add `weatherForecasts: WeatherForecast[]` to both `EmergencyScenario` and `EmergencyDataSnapshot`.
2. **Add Mock Forecasts to Scenarios (`data/mock/emergency-scenarios.ts`)**:
   Add realistic `weatherForecasts` matching stations in Kelantan (e.g. Station: Pasir Mas, Kota Bharu, Tumpat) to the simulated scenarios.
3. **Update Providers**:
   * **Simulation Provider (`data/providers/simulation-provider.ts`)**: Map the forecast array directly from the scenario mock.
   * **Live Provider (`data/providers/live-provider.ts`)**: Include a call to `getWeatherForecasts()` inside the parallel fetch.
4. **Refactor UI to Consume Provider**:
   Remove direct calls to `getWeatherForecasts()` from pages. Instead, retrieve `weatherForecasts` directly from the `getEmergencyData()` snapshot:
   ```typescript
   const data = await getEmergencyData({ mode });
   // Now you can render <WeatherForecastWidget forecasts={data.weatherForecasts} />
   ```

### Phase 2: Consolidate Alerts around the DB-First Pipeline
1. **Transition Dashboard to Normalized Alerts Only**:
   Modify the dashboard and ticker to consume `data.weatherAlerts` instead of mapping raw warnings on-the-fly.
2. **Standardize Components**:
   Ensure components like `LiveUpdateBar` accept normalized alerts or adapt them through a generic mapper, removing direct dependencies on the raw `WeatherWarning` shape.
3. **Audit Edge Function Ingestion**:
   Verify that `sync-live-data` is operating reliably via pg_cron or manual refresh to guarantee that `weather_alerts` database rows are always up to date.
